# step.ml / wrap.ml Prover Port Plan

## Goal

End-to-end **proof-level interoperability** between our PureScript pickles and OCaml pickles: a proof generated by our PS `running_sum` running under `simple_chain` should be bit-for-bit verifiable against OCaml's `simple_chain` verification key, and the inverse.

This is the final gate for the pickles translation effort. Up to this point we have:

- **Byte-equivalent circuits**: `step_main` and `wrap_main` (multi-branch) produce identical constraint system JSON to OCaml fixtures.
- **Advice class machinery**: `StepWitnessM` and `WrapWitnessM` are defined with methods matching the OCaml `Req.*` set.
- **Kimchi FFI**: create/verify proof, oracles, verifier index, etc. exposed through `Pickles.ProofFFI`.

What we do **not** have:

- **Any translation of `step.ml` or `wrap.ml`** ‚Äî the prover orchestration files that bridge runtime inputs (app state, predecessor proofs) to the compiled circuits. The existing `TestContext.purs` infrastructure was written before step_main/wrap_main existed and takes shortcuts that do **not** match OCaml's prover flow ‚Äî it cannot produce byte-interoperable proofs.

## Critical decision: TestContext is not the starting point

`TestContext.purs` does produce proofs and verify them, but it does so via a different orchestration path than OCaml. Functions like `buildStepProverWitness`, `buildWrapProverWitness`, `buildStepFinalizeInput`, `dummyStepAdvice`, `computeStepIvpTranscript` were all invented to make the circuit satisfiable under our own witness derivation. They have no counterpart in `step.ml`/`wrap.ml` and will not produce proofs that match OCaml's.

The correct path forward is **a faithful port of step.ml and wrap.ml**, preserving the exact Fiat-Shamir transcript order, Tick/Tock field assignments, sponge absorption order, and hashing. Anything else cannot pass the interoperability gate.

## Audit findings

Deep audits of both OCaml files were performed (see `mina/src/lib/crypto/pickles/step.ml` and `wrap.ml`). Key findings below; full details in the agent-produced reports available in conversation history if needed.

### step.ml (~900 lines)

One `Make` functor, one top-level `f` function. Inside `f`, the entire prover lives as a cascade of nested helpers plus mutable refs and lazy values. The biggest helper is:

**`expand_proof`** (step.ml:122-537, ~400 lines). For each predecessor wrap proof, derives the advice data the step circuit will read:

1. Graft `app_state` into the proof statement (step.ml:140-148)
2. Convert wire proof ‚Üí kimchi proof (`Wrap_wire_proof.to_kimchi_proof`, step.ml:149)
3. Derive Tick-side `plonk` record (step.ml:150-226) ‚Äî runs `Plonk_checks.Type1.derive_plonk` using `scalars_env` with `Endo.Step_inner_curve.base` and `Common.Max_degree.step_log2`
4. Compute `prev_challenges` via `Ipa.Wrap.compute_challenges` (step.ml:230-235)
5. Call `Wrap_deferred_values.expand_deferred` (step.ml:236-241)
6. Build `prev_statement_with_hashes` including two Poseidon sponge digests:
   - `messages_for_next_step_proof` via `Common.hash_messages_for_next_step_proof`
   - `messages_for_next_wrap_proof` via `Wrap_hack.hash_messages_for_next_wrap_proof`
7. Run `Tock.Oracles.create` (step.ml:304) ‚Äî the **Fq-sponge-based FFI** that replays Fiat-Shamir over the wrap proof
8. Extract oracle outputs: alpha/beta/gamma/zeta (as scalar challenges), v (xi), u (r), sponge_digest_before_evaluations, opening_prechallenges
9. Compute new bulletproof challenges and `b = challenge_poly(zeta) + r * challenge_poly(zetaw)` (step.ml:367-379)
10. Either extract or compute `challenge_polynomial_commitment` via `Ipa.Wrap.compute_sg` (step.ml:380-383) ‚Äî calls kimchi `b_poly_commitment` FFI when `not must_verify`
11. Build `Per_proof_witness.Constant.No_app_state.t` (step.ml:384-410)
12. Derive Tock-side plonk scalars (step.ml:411-463) with a **different** `scalars_env` using `Endo.Wrap_inner_curve.base` and `Common.Max_degree.wrap_log2`
13. Compute `combined_inner_product` (step.ml:464-496) ‚Äî batches `x_hat`, `ft_eval0/1`, all kimchi evals, and the b-polynomial evaluations at zeta and zetaw with xi-weighting
14. Call `Plonk_checks.Type2.derive_plonk` for the Tock side (step.ml:498-511)
15. Wrap result into `Shifted_value.Type2` via `Shifts.tock2` (step.ml:512-536)

**Critical distinction**: step.ml constructs `scalars_env` **twice** ‚Äî once for Tick (`Endo.Step_inner_curve.base` + `Max_degree.step_log2`) and once for Tock (`Endo.Wrap_inner_curve.base` + `Max_degree.wrap_log2`). Mixing these up would typecheck but produce silently-wrong proofs.

**`compute_prev_proof_parts`** (step.ml:547-632) walks the predecessors list, calls `expand_proof` once per prev, and stores results in seven mutable `ref None` slots. Self-recursive vs cross-system predecessors are distinguished via `Type_equal.Id.same self.Tag.id t.id` ‚Äî self uses `self_dlog_vk`/`self_dlog_plonk_index`, external uses the predecessor's `Types_map`-looked-up equivalents.

**Request handler** (step.ml:737-777) dispatches nine `Req.*` tags from the step circuit during witness generation:

- `Compute_prev_proof_parts` ‚Äî runs the expansion loop (must be first)
- `Proof_with_datas`, `Wrap_index`, `App_state`, `Return_value`, `Auxiliary_value`, `Unfinalized_proofs`, `Messages_for_next_wrap_proof`, `Wrap_domain_indices`

**Top-level body** (step.ml:800-853):

```ocaml
let builder = Impls.Step.generate_witness_manual ~handlers:[ handler ] ...
let%bind.Promise res = builder.run_circuit (fun () () -> ...)
let (proof_inputs, next_statement_hashed) = builder.finish_computation res
let proof = Backend.Tick.Proof.create_async ~primary ~auxiliary ~message pk
```

The `~message` parameter is the **previous accumulator** ‚Äî the list of `(sg, challenges)` pairs from predecessors.

### wrap.ml (~580 lines)

Much smaller. Three top-level lets: `challenge_polynomial`, `combined_inner_product` (pure), and a `For_tests_only` module hosting `deferred_values`. Then the top-level `wrap` function.

**`For_tests_only.deferred_values`** (wrap.ml:90-272, ~180 lines). Pure except for one FFI call at the top.

1. `Tick.Oracles.create_with_public_evals` (wrap.ml:99) ‚Äî the **Fp-sponge-based FFI**. Takes `prev_challenges` as `Challenge_polynomial.t` list (wrap.ml:100-105).
2. Extract `x_hat` from `proof.public_evals` or oracle accessors (wrap.ml:108-114)
3. Extract scalar challenges: alpha (SC), beta (raw 128-bit), gamma (raw 128-bit), zeta (SC), joint_combiner (SC, optional), r (= O.u, SC), xi (= O.v, SC)
4. `As_field` module: expand scalar challenges to field elements via `SC.to_field_constant ~endo:Endo.Wrap_inner_curve.scalar` (wrap.ml:133-148)
5. `zetaw = zeta * step_vk.domain.group_gen`
6. `tick_plonk_minimal` (wrap.ml:151-156) ‚Äî with field-form zeta/alpha but raw beta/gamma
7. `tick_combined_evals` via `Plonk_checks.evals_of_split_evals` (wrap.ml:158-164)
8. `tick_domain` via `Plonk_checks.domain` + `Common.tick_shifts` (wrap.ml:165-170)
9. `tick_env` via `Plonk_checks.scalars_env` with `~endo:Endo.Step_inner_curve.base` + `~mds:Tick_field_sponge.params.mds` + `~srs_length_log2:Common.Max_degree.step_log2` (wrap.ml:171-201)
10. `plonk` via `Plonk_checks.Type1.derive_plonk (module Tick.Field) ~shift:Shifts.tick1` (wrap.ml:202-208)
11. `new_bulletproof_challenges`, `b` from `opening_prechallenges` + `challenge_polynomial` (wrap.ml:209-224)
12. Assemble `deferred_values` record (wrap.ml:229-272) ‚Äî overwrites `plonk.zeta`/`alpha`/`beta`/`gamma` with their 128-bit scalar-challenge form (the wrap circuit re-expands them)

**`combined_inner_product`** (wrap.ml:22-62) is pure, operating in `Tick.Field`. Computes `combine(zeta) + r * combine(zetaw)` where each `combine` is Horner batching over `[b_1(pt); ...; b_n(pt); x_hat(pt); ft(pt); w_0(pt); ...; w_14(pt); z(pt); œÉ_0(pt); ...]` with xi weighting. **Exact eval list order must be preserved** (wrap.ml:50-54).

**Top-level `wrap`** (wrap.ml:279-580): similar pattern to step.ml ‚Äî

1. Prepare `messages_for_next_wrap_proof` via `P.Base.Messages_for_next_proof_over_same_field.Wrap.prepare`
2. Build `prev_statement_with_hashes` with `Common.hash_messages_for_next_step_proof` and `Wrap_hack.hash_messages_for_next_wrap_proof`
3. Define request handler for `Evals`, `Step_accs`, `Old_bulletproof_challenges`, `Messages`, `Openings_proof`, `Proof_state`, `Which_branch`, `Wrap_domain_indices`
4. Compute `public_input` via `Common.tick_public_input_of_statement` (for the oracles FFI, NOT the wrap circuit's public input)
5. Call `deferred_values` (pure derivation)
6. Build `next_statement` (the wrap circuit's public input)
7. Assemble `next_accumulator` via `Wrap_hack.pad_accumulator` ‚Äî this is the `~message` arg for the kimchi prover
8. Run `Impls.Wrap.generate_witness_conv ~f:(...)` with the handler installed; inside, call `Backend.Tock.Proof.create_async ~primary ~auxiliary ~message pk`
9. Convert result via `Wrap_wire_proof.of_kimchi_proof`, `Types.Wrap.Statement.to_minimal`

## Fiat-Shamir transcript summary

The critical surface for byte-level interoperability. All absorbs and squeezes must match OCaml exactly.

### Step prover
1. `Common.hash_messages_for_next_step_proof` ‚Äî Tick Poseidon digest
2. `Wrap_hack.hash_messages_for_next_wrap_proof` ‚Äî Tick Poseidon digest
3. `Tock.Oracles.create` ‚Äî Fq-sponge-based FFI (runs inside kimchi Rust; absorbs vk, public input, commitments, evals; squeezes alpha/beta/gamma/zeta/v/u/prechallenges)
4. `Wrap_deferred_values.expand_deferred` sponge ‚Äî Tick Poseidon sequence over `sponge_digest_before_evaluations`, challenges digest, evaluations; squeezes `xi_chal`, `r_chal`

### Wrap prover
1. `Common.hash_messages_for_next_step_proof`
2. `Wrap_hack.hash_messages_for_next_wrap_proof`
3. `Tick.Oracles.create_with_public_evals` ‚Äî Fp-sponge-based FFI. **Takes `prev_challenges` list** ‚Äî the previous wrap proofs' `{commitment, challenges}` pairs. This is a critical FFI extension we need.

## Inventory: what exists in PS vs what we need to port

### REVISED after PS audit (2025-04-09)

The initial inventory below was based on OCaml step.ml/wrap.ml audits without
properly checking what already exists in `packages/pickles/src/Pickles/PlonkChecks/`
and `packages/pickles/src/Pickles/Linearization/`. A deep PS audit revealed that
the PlonkChecks + Linearization infrastructure is much more complete than initially
documented. See the mapping tables below.

#### `scalars_env` ‚Äî MOSTLY EXISTS

`Pickles.Linearization.Env.fieldEnv` (L87-117) produces a pure `Env f` covering
13/18 of OCaml's `scalars_env` fields. The 5 domain-derived values (zkPolynomial,
zetaToNMinus1, zetaToSrsLength, omegaToMinusZkRows, srsLengthLog2) are
**intentionally computed inline** at each FOP call site in both Step and Wrap
`FinalizeOtherProof.purs`, rather than bundled into a single record. This is an
architectural divergence from OCaml (which bundles everything into `Scalars.Env.t`),
but it is functionally complete ‚Äî no mathematical computation is missing.

If a standalone `scalars_env` builder is ever needed (e.g. for the pure prover
layer), it's ~30 LOC of assembly.

#### `ft_eval0` ‚Äî EXISTS (pure + circuit)

- `Pickles.PlonkChecks.FtEval.ftEval0` (L44-53) ‚Äî pure 3-field combiner
- `Pickles.PlonkChecks.Permutation.permContribution` (L106-148) ‚Äî pure perm
  contribution including sigma product, shift product, and boundary quotient
- `Pickles.Linearization.Interpreter.evaluate` (L31-37) ‚Äî pure Polish-notation
  interpreter for gate constraints (the `Sc.constant_term env` analog)
- Both Step and Wrap FOPs inline the full ft_eval0 computation for
  constraint-ordering fidelity

#### `derive_plonk` ‚Äî FUNCTIONALLY EXISTS but not as a named function

`permScalar` (Permutation.purs L86-98, pure) computes the perm scalar.
`zetaToSrsLength` and `zetaToDomainSize` are computed inline at FOP call sites.
`expandPlonkMinimal` (Verify/Types.purs) handles the plonk expansion. The PS
code inlines `derive_plonk` at each FOP call site rather than factoring it into
a standalone function. A standalone `derivePlonk` would be ~40 LOC of assembly.

#### `combine_split_evaluations` ‚Äî EXISTS (circuit)

`Pickles.PlonkChecks.CombinedInnerProduct.hornerCombine` (L156-174) is the
in-circuit Horner fold. `buildEvalList` (L188-197) and `buildEvalListUnmasked`
(L204-218) construct the canonical eval ordering. The two-point combination
(`combine(zeta) + r * combine(zetaw)`) is done inline at FOP call sites.
A pure field-level `combinedInnerProduct` also exists in
`Pickles.PlonkChecks.combinedInnerProduct` (PlonkChecks.purs L55-70).

#### `evals_of_split_evals` ‚Äî NEW PORT (done)

`Pickles.Prove.Pure.EvalsOfSplit.actualEvaluation` (EvalsOfSplit.purs L68-77)
is a net-new port. No PS equivalent existed.

## Key gotchas

From the audit, things that are easy to get wrong and silently produce invalid proofs:

1. **Field crossing in scalars_env**. step.ml constructs `scalars_env` twice with different endo constants: Tick uses `Endo.Step_inner_curve.base` + `Max_degree.step_log2`, Tock uses `Endo.Wrap_inner_curve.base` + `Max_degree.wrap_log2`. These must be paired correctly with the derive_plonk call (Type1 with Tick, Type2 with Tock).

2. **`Endo.Wrap_inner_curve.scalar` vs `Endo.Step_inner_curve.base`**. wrap.ml uses the scalar endo for `SC.to_field_constant` (expanding 128-bit challenges) and the base endo for `scalars_env`. These are different field constants; swapping them is a silent failure.

3. **Vector padding orientation**. Pickles uses `Vector.extend_front_exn` ‚Äî real values at the tail, dummies at the head. Any PS port using `Vector` operations must preserve front-vs-back orientation.

4. **`plonk0.beta/gamma` are raw 128-bit**, NOT endo-expanded. `plonk0.alpha/zeta` ARE endo-expanded. This asymmetry matters for both the sponge path (kimchi treats beta/gamma as already-full-width) and the derive_plonk input.

5. **`combined_inner_product` eval list ordering**: `[b_1(pt); ...; b_n(pt); x_hat(pt); ft(pt); w_0(pt); ...; z(pt); œÉ_0(pt); ...]`. The exact order is `Plonk_types.Evals.to_list e.evals` ‚Äî needs byte-exact replication.

6. **Mutable-ref + lazy pattern in step.ml**. The handler fills refs on the first `Compute_prev_proof_parts` call, and other request responses read those refs via `Option.value_exn`. Porting naively to pure PS will fail; either use `Effect.Ref`, or restructure so all derivations happen upfront and the handler is a closure over the precomputed values.

7. **Order-sensitive `include For_tests_only`** in wrap.ml (wrap.ml:275). The pure `deferred_values` function is defined in a sub-module and brought into `wrap` via `include` ‚Äî an organizational accident, not a design decision. In PS, just live as a top-level helper in the same module.

8. **`messages_for_next_wrap_proof` hash is the chain-of-trust link**. It's absorbed as a public input to the next layer's wrap circuit. Must byte-match OCaml for interoperability.

9. **`actual_proofs_verified` is always 0/1/2**. The pickles invariant limits this; higher values assert-false in OCaml. PS port can assume this bound.

10. **Tick oracles vs Tock oracles**. step.ml calls ONLY Tock oracles (on predecessor wrap proofs). wrap.ml calls ONLY Tick oracles (on the step proof being wrapped). Mixing them up is an easy mistake.
