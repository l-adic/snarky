-- Library API root set for the kimchi package.
--
-- The curated top-level surface the library exists to provide. Lean has no export-list
-- mechanism (everything non-`private` is visible once imported), so this manifest — not the
-- type system — defines the public API. It is the root set for dead-code analysis:
--
--   a declaration in `Kimchi.*` is LIVE iff it is in the transitive constant-dependency
--   closure of these roots; anything compiled but outside that closure is dead.
--
-- Format: one fully-qualified declaration name per line; `--` lines and blanks are ignored.
--
-- Note: the `kimchi-demo` executable (Main.lean) is a SEPARATE build root, reaching the demo
-- defs `Kimchi.Gate.{egQ, egGood, egBad, satisfies}`. Those are demo-only and are not part
-- of this library API surface, so they are reachable through the executable rather than here.

-- The capstone: executable-verifier acceptance ⟹ the index is satisfiable, at every challenge
-- tuple outside the bounded (counting Schwartz–Zippel) bad sets. Combines the Fiat-Shamir
-- junction, the counting quotient soundness, and the IPA special-soundness extraction.
-- `kimchiBundle_sound` is the idealized composition: the transcript bundle is the
-- special-soundness HYPOTHESIS (the tree-of-accepting-transcripts idiom), consumed by one
-- application of the core; the concrete Fiat-Shamir capstone discharges it.
-- `kimchiProof_sound_of_openings` is the openings-interface seam (forking b1): the shared

-- discharge it. Reference side = pure commitment-knowledge; consumer side = per-row openings.

-- The standard-model run-level capstones (grid path): a real accepted run's own sponge
